# Как узнать свой числовой ID пользователя в MAX

Этот ID — то же значение, что в базе и в `.env` в `SUPERADMIN_IDS`: **целое число** (`platform_user_id` для платформы `max`). Его нужно добавить в `SUPERADMIN_IDS`, если вы админите бота именно из приложения MAX (или хотите получать админ-уведомления в MAX).

## Способ 1: логи сервера при входе в бота (основной)

1. Запустите бота с `PLATFORM=max` или `PLATFORM=both` и настроенным `MAX_BOT_TOKEN`.
2. Откройте бота в приложении MAX и отправьте **`/start`** (или нажмите «Старт» в чате с ботом).
3. На сервере в логах процесса бота ищите строки вида **incoming update** / **INCOMING** / **`user_id=`** / **`platform_user_id`**.

В проекте при long polling MAX обновления разбираются в `max_runner`; идентификатор пользователя приходит в поле **`user_id`** внутри события (это и есть ваш числовой ID в MAX).

Если в логах включён вывод текста сообщений, рядом с пользователем часто видно что-то вроде:

- `user=123456789`  
- или структуру JSON апдейта с `"user_id": 123456789`

**Важно:** это не username и не ссылка на профиль — только **число**.

## Способ 2: ответ бота на `/start` (встроенный переключатель, без правки кода)

В проекте уже есть **временный режим**: при включённой переменной окружения бот в MAX **перед** обычным ответом на `/start` присылает отдельное сообщение с твоим числовым **`user_id`** (это `platform_user_id`) и внутренним UUID записи в БД.

Реализация: `max_runner._maybe_max_debug_show_user_id` → вызывается в начале `handle_start`.

### Пошагово

1. **Открой** на сервере (или локально) файл `.env` рядом с процессом бота.
2. **Добавь или измени строку** (для Docker — в `environment` / compose, затем пересобери или перезапусти контейнер):

   ```env
   MAX_DEBUG_SHOW_USER_ID=true
   ```

   Допустимые значения «включено»: `true`, `1`, `yes` (как обычно для boolean в Pydantic Settings).

3. **Перезапусти** процесс бота (чтобы подтянулся новый `.env`).
4. Убедись, что бот запущен с **`PLATFORM=max`** или **`PLATFORM=both`** и задан **`MAX_BOT_TOKEN`**.
5. В приложении **MAX** открой чат с ботом и отправь **`/start`** (или нажми «Старт»).
6. **Первым** придёт служебное сообщение вида «Отладка MAX» с числом в теге `<code>` — это и есть ID для **`SUPERADMIN_IDS`**.
7. **Скопируй** это число в `.env`:

   ```env
   SUPERADMIN_IDS=123456789,987654321
   ```

   (можно добавить к уже существующим Telegram ID через запятую).

8. **Обязательно выключи отладку**, иначе любой пользователь при `/start` увидит свой ID (утечка метаданных, лишний шум):

   ```env
   MAX_DEBUG_SHOW_USER_ID=false
   ```

   или **удали** строку целиком (по умолчанию режим выключен).

9. Снова **перезапусти** бота.

### Зачем в сообщении два значения

- **MAX user_id (число)** — то, что нужно для `SUPERADMIN_IDS` и для понимания логов.
- **UUID в БД** — внутренний `users.id`; для `.env` суперадмина не нужен, только для поддержки и отладки связки аккаунтов.

### Если править код вручную не хочется

Используй только **`MAX_DEBUG_SHOW_USER_ID`** — правки `handle_start` не требуются.

## Способ 3: Telegram и MAX — два разных ID

Если вы заходите и в Telegram, и в MAX:

- в Telegram свой ID можно взять у [@userinfobot](https://t.me/userinfobot) или из логов TG-бота;
- в MAX — только из апдейта MAX (способ 1 или 2).

Оба числа можно перечислить в **`SUPERADMIN_IDS`** через запятую, чтобы права суперадмина и уведомления работали с обеих платформ.

## Где это прописать

В `.env`:

```env
SUPERADMIN_IDS=ваш_telegram_id,ваш_max_id
```

Подробнее см. комментарии в `.env.example`.

## Кабинет business.max.ru

В кабинете разработчика MAX настраиваются **токен бота**, endpoint и режим работы API. **Числовой user_id конечного пользователя** там обычно не показывают как отдельное поле «мой id» — его берут из **входящих webhook/long-poll апдейтов** (способ 1).
