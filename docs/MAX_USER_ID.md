# Как узнать свой числовой ID пользователя в MAX

Этот ID — то же значение, что в базе и в `.env` в `SUPERADMIN_IDS`: **целое число** (`platform_user_id` для платформы `max`). Его нужно добавить в `SUPERADMIN_IDS`, если вы админите бота именно из приложения MAX (или хотите получать админ-уведомления в MAX).

## Способ 1: логи сервера при входе в бота (основной)

1. Запустите бота с `PLATFORM=max` или `PLATFORM=both` и настроенным `MAX_BOT_TOKEN`.
2. Откройте бота в приложении MAX и отправьте **`/start`** (или нажмите «Старт» в чате с ботом).
3. На сервере в логах процесса бота ищите строки вида **incoming update** / **INCOMING** / **`user_id=`** / **`platform_user_id`**.

В проекте при long polling MAX обновления разбираются в `max_runner`; идентификатор пользователя приходит в поле **`user_id`** внутри события (это и есть ваш числовой ID в MAX).

Если в логах включён вывод текста сообщений, рядом с пользователем часто видно что-то вроде:

- `user=123456789`  
- или структуру JSON апдейта с `"user_id": 123456789`

**Важно:** это не username и не ссылка на профиль — только **число**.

## Способ 2: временно вывести ID в ответ бота (если логи недоступны)

Можно на время добавить в обработчик `/start` для MAX отправку сообщения «Ваш MAX user_id: …» (взять `ev.user_id` или `user.platform_user_id` после `get_or_create_user`). После того как запишете ID в `.env`, отладочный ответ лучше убрать.

## Способ 3: Telegram и MAX — два разных ID

Если вы заходите и в Telegram, и в MAX:

- в Telegram свой ID можно взять у [@userinfobot](https://t.me/userinfobot) или из логов TG-бота;
- в MAX — только из апдейта MAX (способ 1 или 2).

Оба числа можно перечислить в **`SUPERADMIN_IDS`** через запятую, чтобы права суперадмина и уведомления работали с обеих платформ.

## Где это прописать

В `.env`:

```env
SUPERADMIN_IDS=ваш_telegram_id,ваш_max_id
```

Подробнее см. комментарии в `.env.example`.

## Кабинет business.max.ru

В кабинете разработчика MAX настраиваются **токен бота**, endpoint и режим работы API. **Числовой user_id конечного пользователя** там обычно не показывают как отдельное поле «мой id» — его берут из **входящих webhook/long-poll апдейтов** (способ 1).
