## 2. Обнаруженные проблемы

### 2.1. Обход проверки подписки (КРИТИЧНО)

**Файл:** `src/handlers/start.py`

**Проблема:** Кнопка «Мотопара» на постоянной клавиатуре (`kb_motopair`) показывает меню «Анкеты Пилотов / Двоек» **без проверки подписки**.

- **ТЗ:** Пользователь без подписки (при включённой подписке) видит только SOS и «Полезные контакты». Функции Мотопары недоступны.
- **Факт:** При входе через `menu_motopair` (inline) проверка есть, через Reply-кнопку «🏍 Мотопара» — нет.
- **Путь обхода:** Пользователь → «Мотопара» → «Анкеты Пилотов» → просмотр анкет и лайки без подписки.

**Рекомендация:** Использовать ту же логику, что и в `cb_motopair_menu`, либо перенаправлять на callback `menu_motopair` с проверкой подписки.

---

### 2.2. Нет проверки подписки для мероприятий

**Файлы:** `src/handlers/start.py` (kb_events), `src/handlers/events.py`

**Проблема:** Пользователь без подписки может просматривать список мероприятий и записываться на них.

- **ТЗ:** При включённой подписке пользователь без неё видит только SOS и «Полезные контакты».
- **Факт:** Список мероприятий, запись и «Ищу двойку/пилота» доступны без проверки подписки.
- **Примечание:** Создание мероприятия уже проверяет оплату, но просмотр и участие — нет.

**Рекомендация:** Добавить `check_subscription_required` в обработчики входа в блок «Мероприятия» (cb_events_menu, kb_events) и при записи на мероприятие.

### 2.3. Блокировка при SOS (частичный bypass)

**Файл:** `src/handlers/middleware.py`

**Статус:** Корректно.

SOS доступен всегда, в том числе заблокированным пользователям — middleware пропускает SOS до проверки блокировки. Соответствует ТЗ.

### 2.5. Ошибка в cb_event_seek_no

**Файл:** `src/handlers/events.py`, ~377

```python
await set_seeking_pair(uuid.UUID(eid), user.id, False)
ev = await get_event_by_id(uuid.UUID(eid))
await callback.message.edit_text(
    _format_event_card(ev),
    reply_markup=get_event_card_kb(eid, True, None),  # eid — строка, OK
)
```

Переменная `eid` уже используется как строка, вызовы с `uuid.UUID(eid)` корректны. Явной ошибки нет, но стоит единообразно использовать UUID или строку.

### 2.7. Тексты сообщений

**Файл:** `src/texts.py`

Тексты соответствуют ТЗ. WELCOME_NEW, REG_ASK_*, SOS_*, MOTOPAIR_* реализованы.  
Не хватает отдельного шаблона «Для доступа к поиску мотопары нужна активная подписка» с перечислением преимуществ, как в примере ТЗ (раздел 10.6).

---

### 2.8. Подписка: источник цен

**Файл:** `src/handlers/profile.py`, `src/services/admin_service.py`

Цены подписки берутся из `get_settings()` (env) и из `SubscriptionSettings` в БД.  
При создании платежа в `cb_subscribe` используется `get_settings()`, а в `cb_profile_raise` — `get_subscription_settings()` (БД). Нужна явная приоритизация (БД > env или наоборот), чтобы не было расхождений.

---

### 2.9. MAX-платформа

**Файлы:** `max_runner.py`, `platforms/max_adapter.py`, `platforms/base.py`

MAX-адаптер частично реализован. Платежи в MAX — в разработке (редирект на веб-страницу). Долгий polling реализован, обработка callback’ов — через `process_max_update`. Полной проверки всех сценариев не проводилось.

## 4. Рекомендации по приоритетам

1. **Высокий приоритет**
   - Исправить обход проверки подписки для Мотопары (kb_motopair).
   - Добавить проверку подписки для входа в блок «Мероприятия» и записи на события.

2. **Средний приоритет**
   - Добавить логирование в пустые `except` в ключевых местах.
   - Уточнить источник цен подписки (БД vs env) и зафиксировать в коде.

3. **Низкий приоритет**
   - Сделать название обязательным для «Масштабных» мероприятий.
   - Добавить возможность указать геолокацию при создании мероприятия (если требуется по ТЗ).
   - Реализовать CRUD городов для суперадмина.

Проект близок к production-ready. Критичных проблем немного; в основном это контроль доступа (подписка) и качество обработки ошибок. После исправлений из раздела 4 и финального тестирования бот может быть выведен в эксплуатацию.
