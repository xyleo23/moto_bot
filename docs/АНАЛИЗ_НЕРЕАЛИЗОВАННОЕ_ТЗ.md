## 2. Полностью не реализовано

### 2.1. Жалобы на мероприятия

**ТЗ 2.3.4 (админ города):** «Просмотр жалоб пользователей на анкеты/**мероприятия** своего города»

**Факт:** Есть жалобы на анкеты (`motopair_report` → уведомление админам). Жалоб на мероприятия нет.

**Нужно:**
- Кнопка «Пожаловаться» в карточке мероприятия
- Модель/таблица для хранения жалоб на мероприятия
- Уведомление city admins + superadmins
- Раздел в админке «Жалобы на мероприятия» (аналогично жалобам на анкеты)

---

### 2.2. CRUD городов

**ТЗ 2.2.14:** «Управление городами (добавление новых городов, назначение админов по городам)»

**Факт:** Город «Екатеринбург» создаётся в `ensure_cities()`. Добавление админов по городу есть. CRUD городов (создание, редактирование, удаление) в админке отсутствует.

**Нужно:**
- Админ-панель: список городов, добавить/редактировать/отключить город
- При добавлении города — автоматическое создание записи в БД
- Редактирование названия, статуса (активен/неактивен)

---

### 2.3. Логи активности

**ТЗ 2.2.11:** «Просмотр статистики и логов (активность, подписки, SOS, мероприятия)»

**Факт:** Статистика реализована (пользователи, блоки, подписки, SOS, мероприятия). Отдельного раздела «логи» (хронология действий, события) нет.

**Нужно:**
- Таблица/хранилище логов: тип события, user_id, timestamp, доп. данные
- Запись: SOS, оформление подписки, создание мероприятия, блокировка и т.п.
- Раздел в админке «Логи» с фильтром по типу, дате, пользователю

---

### 2.4. Шаблоны уведомлений

**ТЗ 2.2.13:** «Управление глобальными текстами: раздел «О нас», **шаблоны уведомлений**, системные сообщения»

**Факт:** Редактируется только `about_us`. Шаблоны уведомлений (при взаимном лайке, при активации подписки и т.д.) в админке не настраиваются — они жёстко прописаны в коде.

**Нужно:**
- Ключи в `global_texts` для каждого шаблона (или отдельная таблица)
- Интерфейс в админке: выбор шаблона, редактирование текста
- Подстановка плейсхолдеров (имя, дата и т.п.) в шаблоны

---

## 3. Частично реализовано / требует доработки

### 3.1. Тексты преимуществ подписки (устарели)

**ТЗ 10.6:** Пример текста «Подписка даёт: … Создание мероприятий бесплатно».

**Факт:** В `profile.py` и `subscription.py` по-прежнему: «Создание мероприятий бесплатно». После последней доработки логика другая:
- Городское мероприятие: платно для пользователя (админ — бесплатно)
- Мотопробеги: 2 в месяц бесплатно с подпиской, далее платно
- Прохваты: бесплатно с подпиской

**Нужно:** Обновить тексты в `cb_profile_subscribe` и при показе «нужна подписка», чтобы отражать реальные условия по типам мероприятий.

---

### 3.2. Настройка лимита мотопробегов в админке

**Факт:** Добавлено поле `event_motorcade_limit_per_month` (по умолчанию 2). В админке его поменять нельзя.

**Нужно:** Добавить в раздел настроек (рядом с event_creation) возможность задать `event_motorcade_limit_per_month` (1, 2, 3…).

---

### 3.3. MAX-платформа

**Факт:** Адаптер MAX, long polling и обработка callback работают. Оплаты через MAX ещё не доделаны (редирект на веб).

**Нужно:** Довести приём платежей в MAX до полноценной поддержки (аналогично Telegram + YooKassa).

## 4. Рекомендации аудита (низкий приоритет)

| Пункт | Статус | Рекомендация |
|-------|--------|--------------|
| Источник цен (БД vs env) | Частично OK | Подписка: БД приоритет, env fallback. Явно задокументировать политику. |
| Пустые `except` | Частично | В критичных местах (рассылка, уведомления) логировать ошибки вместо `pass`. |
| Название масштабного обязательно | OK | В `evcreate_title` уже есть проверка для типа `large`. |
